# v1.13 Share Permission Notes

## Current Finding

The unstable folder permission behavior was not only an SMB timing issue.

The app had UI and saved data for per-friend `指定`, but the actual Windows ACL application only handled:

- 全員
- 読みのみ
- 共有OFF

Because all remote friends currently use the same `swkguest` credential, Windows cannot honestly distinguish "NEC only" from "Win11 only" using the current single-key design.

## v1.13 Grounding Change

v1.13 intentionally limits the live permission surface to:

- 全員
- 読みのみ
- 共有OFF

Per-friend `指定` is shown as a future/key-design item rather than pretending it is enforced.

Old `permissions.json` user names are ignored on load and save for live ACL purposes.

## Why

This avoids the dangerous state where:

- the UI says `指定`,
- `permissions.json` contains a friend name,
- but Windows still grants access through the shared `swkguest` account.

## Next Real Design Step

To make true per-friend sharing work, ShareWorkin likely needs per-friend credentials or an app-mediated access model.

Do not solve this with notification/reload behavior alone. Notifications and reload buttons can help after the state model is correct, but they should not hide an unenforced permission model.

## 2026-05-08 Follow-up: Shop Ownership And ACL Alignment

User report: on the right-side PC, the Toshiba shop showed most items but hid `Desktopが作成`. Logs showed the folder was enumerated at the shop root but became `isSharedOff=True` because remote `swkguest` could not read it (`win32err=5`). Direct UNC access also returned access denied.

Important conclusion: this is not acceptable as "a brought-in folder cannot be shared." When the owner PC runs `共有を開く`, the selected shop folder and its children should become manageable as that user's shop. If needed, the app should align ownership internally and then apply the intended access rule.

Implemented in commit:

- `93607fd fix: お店配下のアクセス設定を開店時に整える`

Behavior after this commit:

- Opening a shop checks the root and children for ACL repairability, not only the root.
- If repair requires taking ownership, the user-facing wording says "アクセス設定を整える" rather than exposing Windows ownership jargon.
- After consent, the shop tree ownership is aligned to the current Windows user before ShareWorkin ACLs are applied.
- Reapplying permissions during shop open is synchronous before the list is shown.
- `全員` resets the target and descendants to inherited ACLs.
- `読みのみ` and `共有OFF` set the parent rule, then reset children to inherit the parent rule.
- Permission changes through `許可指定` also first align ownership for the target file/folder.
- Items placed, moved, renamed, or externally created while the shop is open go through `SharePolicyRepair`, which aligns ownership and resets inherited ACLs.
