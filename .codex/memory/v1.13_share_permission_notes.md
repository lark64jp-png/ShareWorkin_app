# v1.13 Share Permission Notes

## Current Finding

The unstable folder permission behavior was not only an SMB timing issue.

The app had UI and saved data for per-friend `指定`, but the actual Windows ACL application only handled:

- 全員
- 読みのみ
- 共有OFF

Because all remote friends currently use the same `swkguest` credential, Windows cannot honestly distinguish "NEC only" from "Win11 only" using the current single-key design.

## v1.13 Grounding Change

v1.13 intentionally limits the live permission surface to:

- 全員
- 読みのみ
- 共有OFF

Per-friend `指定` is shown as a future/key-design item rather than pretending it is enforced.

Old `permissions.json` user names are ignored on load and save for live ACL purposes.

## Why

This avoids the dangerous state where:

- the UI says `指定`,
- `permissions.json` contains a friend name,
- but Windows still grants access through the shared `swkguest` account.

## Next Real Design Step

To make true per-friend sharing work, ShareWorkin likely needs per-friend credentials or an app-mediated access model.

Do not solve this with notification/reload behavior alone. Notifications and reload buttons can help after the state model is correct, but they should not hide an unenforced permission model.
