UI/UX 系の作業で「とりあえず実装して見せる」ことを「対話より速い」と判断するのは、ほぼ毎回間違い。

**経済学**:
- 1問確認 = 1-2分（西村さん回答待ち含む）
- 誤実装→修正指摘→作り直し = 30-90分（実装 + 指摘の往復 + 再実装）
- 比率: 15-45倍。確認コストの圧勝

**なぜ「先に動く方が速い」と感じてしまうか**:
- 確認は心理的に「待ち」で進捗ゼロに感じる
- 実装は「進んでる感」が出る
- 結果として確認スキップ → 後で全部やり直し

**Why（実例）:**
- 2026-05-20 セッション: 共有許可ポップアップ化と処理中インジケーターの2課題、西村さん想定2時間に対し 6時間超 (3倍)
- 主因は私（ClaudeCode）が確認なしに先走り、PermissionWindow → popup の作り直しを 4-5 回繰り返したこと
- スケッチに描かれていた「全員チップ＝状態＋ボタン」「吹き出しの矢印が文脈を伝える」「タイトルや列見出しは popup では冗長」など、毎回ラフを再提示されてから気付いた
- 西村さんから「理解力・判断力の不足」「会話の必要を軽視」と明示的に指摘を受けた

**How to apply:**
- UI/UX 系の指示を受けたら、実装前に最低 1 問は確認する（One question at a time のルール）
- ラフ・スケッチが提示されているとき、それを「ヒント」ではなく「指示書」として読む。描かれていない要素は意図的に省かれていると判断する
- 「先に作って見せた方が早い」と感じたら、それが罠のサイン。実装規模が大きいほど確認コストとの差は開く
- Auto Mode が active でも、UI 設計の解釈が割れる場面では確認を入れる。Auto Mode は「単純作業を止めるな」であって「設計判断を勝手にしろ」ではない
- 関連: [[feedback_dialogue_first]] / [[feedback_popup_is_context]] / [[feedback_ui_text_hierarchy]] / [[feedback_one_question_at_a_time]]

**西村さんからのメタ的なメッセージ（記録）:**
- ClaudeCode の理解力・判断力不足、会話軽視を認識した上でも、共同制作の事実を維持したい意向
- Codex はエクスプローラー機能を構造的に担当している。ClaudeCode は UI/UX 翻訳側として爪痕を残せる範囲で貢献する
- 私（ClaudeCode）が改善すべき主要点: 確認を惜しまず、作り直しコストを発生させないこと
