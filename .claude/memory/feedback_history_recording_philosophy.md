## 原則：それぞれがそれぞれの必要で履歴化する

更新履歴の記録方針は2軸。

**軸1: 網羅性（Coverage）**
ある操作が引き起こした「全ての状態変化」を記録する。直接の動作だけでなく、副作用（権限継承・パーミッション変化）も含む。

**軸2: 密度（Density）**
利用可能なフィールドをできる限り埋める。targetName / pathText / sourcePath / destinationPath / note / source / outcome すべてに意味ある値を入れる。

## 「全部履歴に出す」と「大量エントリを避ける」は矛盾しない

- 「全部出す」= 状態変化を一切スキップしない
- 「大量を避ける」= 設計の粒度で解決する。記録を省くことで避けるのではない

例: フォルダー内の大量ファイルへの PermissionCascade → 各ファイルを記録する仕様だが、「大量になる事象をまず作る → そこからどうすると有効か考える」という順序で設計する。事象を経験せずに設計を決めるな。

## 機械的適用の禁止

「前そう言われた」という気持ちで実装してはいけない。各シナリオについて「なぜこれを履歴化するのか」を自分で考えてから実装する。理由のない記録は後で必ず迷走する。
