# UI 微修正の基本スタンス

2026-05-16 時点の合意。

## 前提

- 目標は「Google Drive 風」ではなく、**Windows エクスプローラー準拠**を基本にする
- Google Drive は説明しやすくするための参考であり、仕様の主基準ではない
- 利用者からの依頼は、構造説明よりも**使った感覚・違和感・主観**ベースで来る前提で受ける

## この種の依頼の受け方

- 利用者目線の報告をそのまま有効な入力として扱う
- 「どの画面で / 何をしたら / どうなったか / 本当はどうなってほしいか」で十分
- 実装構造の理解は Codex 側で引き受ける。専門用語を要求しない
- 必要なときだけ、選択肢を狭める短い確認を行う

## 修正の進め方

- 重い一括改修ではなく、**1テーマずつ**直す
- 毎回の流れは `現象整理 -> 影響範囲を限定 -> 小変更 -> build確認 -> 次の気づきへ`
- 複数 PC をまたぐ更新インストールや共有設定のような大作業ではなく、軽い反復前提で進める
- VS Code 再起動前提の進行にしない。レスポンス重視

## 守るべき方針

- 既存機能への被害拡大を最優先で避ける
- まず構造を整えてから、小さな修正を積む
- UI 部品の総入れ替えより、既存部品の局所修正を優先する
- 挙動変更と構造整理はなるべく分ける

## 現在の構造上の足場

- `ShopItemsListView` の一覧操作は、今後触りやすいように [MainWindow.ShopItemsInteractions.cs](../../ShareWorkin/MainWindow.ShopItemsInteractions.cs) へ分離済み
- 一覧操作の調整は、まずこのファイルを起点に検討する
- `MainWindow.xaml.cs` 本体へ広く散らして修正しない

## 一覧操作の基準

- クリック、Ctrl+クリック、Shift+クリック、ダブルクリック、右クリック、空白クリック、ドラッグ開始、空白ドラッグを Explorer 基準で考える
- 独自挙動がある場合は、まず Explorer 基準からのズレとして整理する
- 最初に疑うべき箇所は、選択・ドラッグ・リネーム開始が密結合している部分

## 新規セッション開始時の扱い

- まずこのメモと `project_v1.18_ui_dragdrop_session.md` を確認する
- 利用者から来る「ここが変」「ここだけ直したい」を歓迎し、小さく受ける
- 大きな設計変更を提案する前に、軽い反復で解けるかを先に検討する
