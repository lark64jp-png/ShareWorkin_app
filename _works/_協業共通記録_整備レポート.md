# 協業共通記録_整備レポート

この文書は、ShareWorkin の共有エクスプローラー構造刷新における協業の正本である。  
過去の経緯を積み上げる場所ではなく、「今この時点で何を正とするか」を明示するために使う。

## 最重要注意

- この文書を先に読み、ここに書かれた前提を正として扱う
- 過去のレポートは必要になったときだけ参照する
- `MainWindow` に explorer 操作本体を戻さない
- 実装は段階移行とし、全面書き換えはしない
- 成功だけでなく、失敗・ブロック・観測ログも履歴対象にする

## 今の正解

### 構造方針

- `MainWindow` は UI 制御中心に寄せる
- 操作の事前判定と実行は `ExplorerActionService` に寄せる
- 操作結果は共通結果モデルで返す
- 後処理は操作本体から分ける
- watcher / polling 由来の変化も最終的に同じ履歴モデルへ流す

### 履歴方針

- 更新履歴の中心は追記型ジャーナルで育てる
- 移行中は `HistoryRepository` を読み出し窓口として残してよい
- `Update` は `history.json` とジャーナルの橋渡しを暫定的に許容する
- この二重構造は恒久運用にしない

#### 記録の2軸原則（2026-05-18 確立）

**網羅性（Coverage）**: ある操作が引き起こした「全ての状態変化」を記録する。直接の動作だけでなく、副作用（権限継承・パーミッション変化など）も含む。

**密度（Density）**: 利用可能なフィールドを極力埋める。`targetName` / `pathText` / `sourcePath` / `destinationPath` / `note` / `source` / `outcome` すべてに意味ある値を入れる。

「全部履歴に出す」と「大量エントリを避ける設計」は矛盾しない。大量になる事象はまず作って、そこからどうすると有効な履歴になるかを考える順序で設計する。記録を省くことで大量を避けるのではない。

履歴に出ることを完了基準にする——この基準が機能するのは、記録が正しく出ている状態が前提である。

### 更新履歴フォーム方針

- 今回の更新履歴フォームは、完成形を意識しつつも「修正と状況把握」を優先する
- 構成は `表示部` と `ディティール部` の二層にする
- 表示部は利用者目線の要約一覧とする
- ディティール部はログ的・調査的・プログラム視点の確認場所とする
- 一覧で足りない情報は、捨てずにディティール部へ集約する

### 協業方針

- Codex は骨格、境界、共通記録を先に整える
- ClaudeCode には、まず構造理解・レビュー・限定範囲の実装を依頼する
- 触る層を先に決めてから作業する

## 現在の状態

- `SwkHistoryJournal` は追加済み
- 更新履歴はジャーナルへ寄せ始めている
- 更新履歴ウィンドウは開いたまま追記確認できる
- 更新履歴ウィンドウは `表示部 + ディティール部` の方向へ着手済み
- 更新履歴ウィンドウは正本定義の表示列を反映済み
- 更新履歴ウィンドウのディティール部は実装済み
- 更新履歴ウィンドウに `削除基準日時` 入力欄と `履歴削除` ボタンを追加済み
- 空欄時の全削除確認と、指定日時以前の履歴削除を実装済み
- 削除は表示だけでなく保存元にも反映される
- `ExplorerActionService` は追加済み
- `移動`（単体・複数）`名前変更` `コピー` `外部配置` `保留` `削除` `フォルダー作成` を共通サービス化済み
- 成功・失敗・ブロックの全結果がジャーナルへ記録される経路が整った
- パス情報（移動元・移動先・旧名など）もジャーナルに流れるようになった
- watcher（`ArrivalSensor_Created` / `ArrivalSensor_Renamed`）と polling（`PollingTimer_Tick`）起因のイベントが履歴モデルへ接続済み
  - `source: "Watcher"` / `source: "Polling"` で出どころを区別
  - 自操作由来（suppress 中）の外部リネームは二重記録を除外
  - パス情報（pathText / destinationPath / destinationFolder / sourcePath）もジャーナルに流れる
- 移動時のパーミッション変化を `PermissionChanged` として記録済み（移動前後ステータス・移動先フォルダー名）
- コピー・外部配置時の権限継承を `PermissionChanged` として記録済み
- フォルダー権限変更時の `PermissionCascade` を再帰化済み（`SearchOption.AllDirectories`・per-item の pathText）
- 友達削除に `Delete` 履歴を追加済み
- Switch / RefreshFriend / Register / UpdateFriend / 接続・接続失敗操作に `pathText` と `source` を追加済み
- 行選択色をアクティブ・非アクティブで統一済み（`#C8D8F0`）
- アクション訳語に `PermissionChanged` / `PermissionCascade` / `Copy` / `Log` を追加済み
- `history.json` / `history-journal.jsonl` をインストーラー更新引き継ぎ対象に追加済み

## 表示部の正本定義

- `日時`
- `ユーザー`
- `パス`
- `ファイル名`
- `アクション`
- `内容`
- `備考`

### 表示部の意味

- `日時`: いつ起きたか
- `ユーザー`: 自分または関与相手
- `パス`: 結果重視の場所。移動なら移動先、名前変更なら変更後の置き場所
- `ファイル名`: 対象名。ファイルでもフォルダーでもここへ入れる
- `アクション`: 操作種別
- `内容`: 何をどうしたかの利用者向け短文
- `備考`: 元の場所、旧名、失敗理由、ブロック理由などの補足

## ディティール部の正本方針

- 表示部と同じ出来事を、より内部的に確認できることを重視する
- 少なくとも次の情報を保持・表示対象に育てる

- `OccurredAt`
- `Source`
- `EventType`
- `Outcome`
- `TargetName`
- `PathText`
- `Message`
- `Note`
- `SourcePath`
- `DestinationPath`
- `DestinationFolder`
- `FriendId`
- `FriendName`
- `Direction`
