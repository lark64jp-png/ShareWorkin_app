# Codex向け_ログGit確認ZIP作成依頼_2026-05-26

## 目的

この依頼は、すぐに追加修正を行うためのものではありません。

ShareWorkinの完成直前段階では、実機テストで見える現象だけではなく、ログ・Git履歴・変更箇所の偏りから、同じ場所で同じ種類のミスを繰り返していないかを確認する必要があります。

そのため、ChatGPT監修に渡せる確認用ZIPを作成してください。

目的は以下です。

- レギュラー的なログチェック視点を確立する
- Git履歴から、進行状況・修正の偏り・同じ場所を繰り返し触っている危険を確認する
- Codexが局所修正に寄りすぎていないかを外部監修で確認する
- 実装済み内容・未解決内容・テスト結果・ログ設計を一式で確認できる状態にする

## 重要ルール

この依頼を受けても、追加実装・修正・整理変更は行わないでください。

まずは、現状をそのまま確認用ZIPとしてまとめてください。

「ついでに直す」「見つけたので修正する」「ログを追加してから渡す」は禁止です。

必要な修正案がある場合は、ZIP内のレポートに「提案」として書くだけにしてください。

## ZIPに含めてほしいもの

### 1. 現状レポート

ファイル名例：`現状レポート_ログGit確認用.md`

内容：

- 現在の作業目的
- 直近で修正した内容
- まだ不安が残る箇所
- Codexとして気になっている箇所
- 追加修正に入る前にChatGPTへ確認してほしい点
- 今回は実装変更ではなく監修確認用ZIPであること

### 2. Git履歴レポート

ファイル名例：`Git履歴レポート_直近変更確認.md`

内容：

- 直近10〜20件程度のコミット一覧
- 各コミットの目的
- 変更ファイル一覧
- 同じファイル・同じ機能を繰り返し触っている箇所
- 迷走・手戻り・再修正が疑われる箇所
- Codex自身の見立てとして、修正が集中している危険箇所

可能なら以下も含めてください。

```bash
git log --oneline --decorate -n 20
git diff --stat HEAD~5..HEAD
git diff --name-only HEAD~5..HEAD
```

ただし、出力結果をそのまま貼るだけでなく、Codexの短い解釈も添えてください。

### 3. 変更差分レポート

ファイル名例：`変更差分レポート_監修用.md`

内容：

- 直近修正で変更した主要ファイル
- 各ファイルで何を変えたか
- 何を変えていないか
- 相手認識、Tray通知、ファイル受信、ログ出力、インストーラー作成に関係する変更有無
- 影響範囲の見立て

ChatGPT監修では、ここを見て「関係ない修正が混ざっていないか」「同じ場所を再修正していないか」を確認します。

### 4. ログ設計レポート

ファイル名例：`ログ設計レポート_異常検出視点.md`

内容：

ログを単に増やしたかではなく、何を検出するためのログかを整理してください。

最低限、以下を項目化してください。

- 相手認識ログ
- ファイル受信ログ
- ファイル送信ログ
- Tray通知ログ
- 証明書/TLS照合ログ
- 自動補正ログ
- 手動確認ログ
- UI/履歴へ渡した最終判定ログ

各項目について、次を記載してください。

- 何を検出するためのログか
- 正常時の出力イメージ
- 異常時の出力イメージ
- 成功して見えるが内部的に怪しいケース
- そのログを見たときの判断基準

### 5. 実際のログサンプル

ファイル名例：`ログサンプル_直近テスト.md`

可能な範囲で、実際に出力されたログを含めてください。

含めてほしいパターン：

- 正常送信
- 正常受信
- 相手認識成功
- 相手認識が高確度候補になるケース
- 未照合になるケース
- 証明書不一致または通知失敗のケース
- 自動補正が走ったケース
- 自動補正を見送ったケース

実ログがまだないパターンは、無理に作らず「未取得」と明記してください。

### 6. テスト手順と判定基準

ファイル名例：`テスト手順_成功失敗判定.md`

内容：

- 実機テスト手順
- 成功判定
- 失敗判定
- 失敗時に見るログファイル
- 検索キーワード
- 正常時との差分
- 次に判断すべきこと

「ログを見てください」だけではなく、どのログの何を見るかまで書いてください。

### 7. 最終インストーラー同一性確認

ファイル名例：`インストーラー同一性確認.md`

内容：

- 現在のソース状態
- GitコミットID
- BuildTime
- InstallerBuildTime
- AppVersion
- どのインストーラーが最新か
- そのインストーラーに今回の修正が含まれる根拠
- ユーザーが実機で確認できる表示またはログ

過去に「最終インストーラーかどうか」が曖昧になったため、ここは必須です。

### 8. 関連ソース

以下に関係するソースを含めてください。

- 相手認識
- 登録利用者プロファイル
- エクスプローラー表示
- ファイル送信
- ファイル受信
- Tray通知
- TLS/証明書
- ログ出力
- 履歴/UI表示
- インストーラー作成スクリプト

ソース一式でも構いませんが、可能なら上記関連部分が分かるように整理してください。

## ChatGPTに見てもらいたい観点

ZIP作成後、ChatGPTには以下を確認してもらいます。

- ログチェック視点が実用的か
- 成功して見える裏側の異常を拾えるログになっているか
- Git履歴から、同じ場所で同じ種類のミスを繰り返していないか
- 修正が場当たり的に継ぎ足されていないか
- 相手認識仕様が、登録・表示・送信・受信・通知で分散しすぎていないか
- 最終インストーラーとソースの同一性が確認できるか
- 次の修正に入ってよい状態か

## Codexへのお願い

今回のZIPは、実装成果物ではなく、監修用の判断材料です。

したがって、見栄えのよい説明よりも、後から検証できる具体情報を優先してください。

特に以下を重視してください。

- どこを見ればよいか
- 何を根拠に判断したか
- 何が未確認か
- どこに不安が残るか
- 何をまだ触っていないか

以上をまとめたZIPを作成してください。
