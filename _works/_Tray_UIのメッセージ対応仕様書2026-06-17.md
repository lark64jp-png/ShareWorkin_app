## 5. 状態定義の逆引き

`ShareWorkin/MainWindow.xaml.cs` 45〜54行付近。

```csharp

{
    Hidden,
    WaitingMarkOnly,
    WaitingMessage,

    TimedOut,
    InternalError,
}
```

### 5.1 実装上の状態意味

| 状態 | UI意味 |
|---|---|
| `Hidden` | 通常時。行はあるが中身なし |
| `WaitingMarkOnly` | リクエスト開始直後。`●` のみ |
| `WaitingMessage` | 400ms超過後。`● 共有状態を確認しています...` |
| `StaleFallback` | 最新ではなく前回情報・代替応答を表示 |
| `Failed` | 取得失敗系 |
| `TimedOut` | タイムアウト |
| `InternalError` | 内部例外 |

### 5.2 当初設計との差分

当初案では `ErrorAccessDenied`, `ErrorMismatch`, `ErrorGeneric` などを状態enumで分ける案でした。  
現在実装では、状態enumは簡略化され、エラー内容の違いは `TrayUiErrorReason` と表示メッセージ側で吸収しています。

これは実装簡略化としては許容できます。  
ただし、将来的にUI色やボタン出し分けを細かく変える場合は、状態enumを再分化する余地があります。

## 9. Tray応答からUI表示への変換逆引き

`ShareWorkin/MainWindow.xaml.cs` 10259〜10329行付近。

```csharp
private void ApplyTrayCommandResponseToStatusBar(
    TrayCommandUiResponse response,
    TrayRetryRequest? retryRequest = null)
```

### 9.1 変換表

| ResultCode | UI表示 | 再確認 |
|---|---|---|
| `Success` | 非表示へ戻す | なし |
| `FallbackSuccess` | 「前回確認できた状態を表示しています。」またはDetail | Retryableなら表示 |
| `TimedOut` | 「共有状態の確認がタイムアウトしました。」またはDetail | Retryableなら表示 |
| `InternalError` | 「共有状態の確認中に内部エラーが発生しました。」またはDetail | Retryableなら表示 |
| `Failed` + `AccessDenied` | 「共有状態を確認する権限がありません。」 | Retryableなら表示 |
| `Failed` + `TargetMismatch` | 「共有状態が現在の接続先と一致しません。」 | Retryableなら表示 |
| `Failed` + `Cancelled` | 「共有状態の確認を中止しました。」 | Retryableなら表示 |
| `Failed` + `TrayUnavailable` | 「ShareWorkinTray に接続できませんでした。」 | Retryableなら表示 |
| `Failed` + その他 | 「共有状態を確認できませんでした。」またはDetail | Retryableなら表示 |

### 9.2 重要点

`ApplyTrayCommandResponseToStatusBar(...)` の冒頭で `CompleteTrayStatusBarRequest()` を呼んでいます。  
これにより、成功・失敗を問わず、リクエスト中表示は解除されます。

### 9.3 未整理点

`Cancelled` が `Failed` に含まれているため、現在はキャンセル時にも再確認ボタンが出る可能性があります。  
実害は限定的ですが、将来の共通応答化時には `Cancelled` を静かに扱うかどうかを再整理する余地があります。

## 14. 現時点での仕様判定

### 採用できる点

- MainWindow共通状態バーの配置は妥当。
- 既存のAdminWorker/StatusText/ProcessingBarへ混ぜない方針は守られている。
- 行常設・中身切替の方針も守られている。
- `●` の一瞬表示と400ms後のメッセージ表示が入っている。
- 応答後に成功/失敗を問わず進行中表示を解除する仕組みがある。
- retryable応答から再確認へつなげる土台がある。

### 次に修正・確認したい点

1. Tray未接続時の表示は、現状コードでは共通状態バーへ寄せ済み。
2. `if (false && ...)` の無効分岐を削除する。
3. 文字化けコメントを削除する。
4. `CompleteTrayStatusBarRequest()` の呼び出し責務を整理する。
5. リクエスト中マークの色を当初方針どおり橙系にするか判断する。
6. `FallbackSuccess` の文言を用途別に分ける。
7. 次段階で `GET_SHARE_SNAPSHOT` を追加し、今回の共通バーへ接続する。

## 15. 次段階の実装対象

今回の土台実装後に進めるべき対象は以下です。

1. TrayPipeServer側に `GET_SHARE_SNAPSHOT` を追加する。
2. `Get-SmbShare` / `Get-SmbShareAccess` 相当からWindows共有定義を取得する。
3. Tray内でWindows確認済みsnapshotを保持する。
4. 同一ShareName/ShopRootPathへの問い合わせを1本に合流する。
5. `TrayCommandResponse<TPayload>` の正式JSONを追加する。
6. UiPipeClient側をrequestId付き応答に対応させる。
7. MainWindow側で `GET_SHARE_SNAPSHOT` 応答を `ApplyTrayCommandResponseToStatusBar(...)` へ流す。
8. エクスプローラー表示更新入口10系統を共通問い合わせへ寄せる。

`ApplyTrayCommandResponseToStatusBar(...)` へ流す共有状態確認は、通知テストだけではなく、ローカル共有状態確認にも接続済みです。

現時点で接続済みの入口:

- 起動直後
- 自分の共有への切り替え時
- リロードボタン
- ローカル側のフォルダー移動 / 戻る / 進む / ルート移動
- 共有開始成功後
- 共有終了時
- Explorer 操作成功後
- 権限変更成功後
- 遅延再表示
- `ArrivalSensor` の `Created` / `Deleted` / `Renamed` / `Error`
- `ContentsSensor` の `Created` / `Deleted` / `Changed` / `Renamed` / `Error`

### 17.4 UI 側の追加制御

- `QueueLocalShareSnapshotRefreshAsync(...)` を追加し、UI 側でも多重呼び出しを 1 本の更新ループに合流
- `RequestLocalShareSnapshotStatusRefresh(...)` からは直接取得せず、上記キューへ積む方式に変更
- 非 `forceRefresh` の再確認は短い debounce を入れ、watcher / 連続操作由来の密集呼び出しをまとめる
- ローカル共有確認中に相手共有へ切り替わった場合、古い応答を UI へ適用しないための文脈一致確認を追加
- 相手共有へ遷移した時、相手閉店通知受信時、相手オフライン確定時はローカル共有用状態バーを `Hidden` へ戻す

### 17.5 この時点でも未整理の項目

- 本文中の初版説明には、通知テストだけが適用点であるかのような古い説明が残っている
- 初版時点の「未実装」記述は、履歴としては残っているが現状コードとは一致しない箇所がある
- 実機確認後、watcher 由来の再確認頻度が高すぎる場合は追加の間引き調整余地あり
