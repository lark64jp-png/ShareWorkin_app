# 通知設定確認経緯レポート 2026-06-04

## 経緯概要

通知設定パネルで「通知状態: テスト送信済み」「Windows通知ON」に見える状態でも
「テスト通知を送る」ボタンの強調表示が残っている可能性があるとの問題意識から調査を開始した。

---

## 今回追加したログ項目

`MainWindow.xaml.cs` の `LogNotificationSupportSnapshot` メソッドに以下3項目を追加した。

| 項目 | 内容 |
|---|---|
| `shareWorkinNotificationsEnabled` | Windows通知設定のShareWorkin個別ON/OFFのレジストリ取得結果 |
| `isAttentionNeeded` | ボタン強調のON/OFF判定結果 |
| `attentionReason` | 強調ONの場合の理由（`WindowsOff` / `ShareWorkinOff` / `ModeOff` / `Unverified` のいずれか、またはカンマ区切りの複数） |

強調なしの場合は `attentionReason=none` と出力される。

ログはパネル開閉時・「Windows通知設定を開く」クリック時に出力される。

---

## デバッグ版での確認結果と判断

デバッグ版（`H:\ShareWorkin_app\repo\build\ShareWorkin\bin\Debug\net8.0-windows\ShareWorkin.exe`）を
起動してパネルを開いた結果、以下のログが得られた。

```
windowsNotificationsEnabled=True
shareWorkinNotificationsEnabled=True
isAttentionNeeded=True
attentionReason=Unverified
notificationMode=All
supportState=Unverified
lastTestAt=null
```

`attentionReason=Unverified` だったが、これはデバッグ版が独立した settings.json を持つためである。

settings.json の保存場所はコード上 `AppContext.BaseDirectory`（起動した exe のあるフォルダー）で決まる。
デバッグ版は初回起動のため settings.json が存在せず、NotificationSupportState が初期値の
`Unverified`・lastTestAt が null になった。

**Windows通知取得（原因B）は正常。`windowsNotificationsEnabled=True`、`shareWorkinNotificationsEnabled=True` を確認。**

## 次の確認方針

### 方針

ログ追加済みビルドをインストーラー化し、インストール版環境（`C:\MyApps\ShareWorkin\`）で
パネルを開いた後のログを確認する。

### インストーラー経由で確認する理由

- `C:\MyApps\ShareWorkin\` を手作業でファイル上書きすると、インストーラーが管理する構成と実態がずれる
- インストーラー経由であれば §E の「削除してから入れる」原則に沿い、クリーンな状態で確認できる
- 手作業上書きで確認した結果は、インストーラー経由の動作と一致するとは限らない
- Git番号とインストーラー・実機動作の対応関係を保つために、正規の手順で確認することが望ましい

### 確認ポイント

インストール版起動後のログで以下を確認する。

```
shareWorkinNotificationsEnabled=True/False
isAttentionNeeded=True/False
attentionReason=???
```

`attentionReason` が `none` であれば、強調は解除されており問題なし。
`attentionReason` に値があれば、その内容に応じて修正方針を決定する。
