**起草:** ClaudeCode（草案8初稿）／リライト方針に基づき再構成

## 本書の前提

- 本書は「よくある Windows アプリの作法」をなぞるための文書ではない
- 本書は、私が目指す「便利アプリ」のブランドルールを、配布・更新・削除・復元まで含めて定める文書である
- このブランドで配るアプリは、利用者に「入れやすい」「わかりやすい」「消しやすい」「やり直しやすい」を約束する
- 便利さは UI だけでは成立しない。インストール、更新、削除、移動、復元で不安を与えないことも製品品質に含む
- **インターネット経由で不特定利用者へ配布することを前提とする。動作品質だけでなく、利用者への説明責任も制作対象とする**
- 書面は守られないことがある。したがって、本書は「理念」だけでなく「破れやすい箇所」も明示する
- **本書は絶対原則集ではない。実装・運用・実機観測を優先する。書面と実態が矛盾した場合は書面を見直す。基準書は判断を揃えるための道具であり、思考停止の根拠にしてはならない**

---

## 用語

- **私** または **あなた** ─ 指示者であり、ブランドオーナー
- **利用者** ─ 配布アプリを使う人
- **アプリホルダー** ─ 当該アプリの exe、設定、秘密、履歴、利用者保管物を収めるフォルダー
- **インストールルート** ─ 原則 `C:\MyApps\`。ただし本質は「固定パス」ではなく「利用者から見てわかる移動可能ロケーション」
- **Layer 1** ─ アプリ自身の自己完結
- **Layer 2** ─ インストーラー/アンインストーラーの能動的一掃
- **System 依存物** ─ SMB 共有、ACL、Firewall、スケジュールタスク、ローカルユーザー、DPAPI 等、Windows 側に保持されるもの
- **復元起点** ─ 作業を再開するために戻す基準状態。原則は「共有ホルダー + アプリホルダーのバックアップ」で再現できる状態を指す
- **説明責任** ─ 利用者が制作者・保存先・削除影響・更新元を自分で確認できる状態を維持すること

## §A. ブランド原則

- このブランドのアプリは、利用者に「アプリの居場所」が見えることを重視する
- このブランドのアプリは、利用者に「どこに何があるかわからない」状態を作らない
- このブランドのアプリは、利用者に「消えたと思ったのに残っている」状態を作らない
- このブランドのアプリは、利用者に「再開したいのにどこまで戻せばよいかわからない」状態を作らない
- よって、**保存先・削除対象・復元対象は、配布時点で説明可能でなければならない**
- このブランドは、**動作品質と利用者への説明責任の両方を設計対象とする**
- 利用者が「誰が作ったか・どこに何があるか・消したら何が消えるか・更新はどこから来るか」を自分で確認できる状態を作ることは、製品品質の一部である

---

## §B. ホルダー自己完結(Layer 1)

- アプリが恒常的に使うデータは、原則として **すべてアプリホルダー内** に置く
- 対象は、設定、秘密、履歴、ログ、保留、カスタム画像、キャッシュ、証明書、利用者が認識する補助状態を含む
- 「起動後に増えるデータだけ外に出る」は禁止する
- `AppData`、`ProgramData`、レジストリ書き込み、ユーザープロファイル配下の恒常保存は原則禁止
- 旧版移行のための `LocalAppData` 参照は許容するが、**現役の保存先として使ってはならない**
- アプリホルダーは、**exe からの相対関係で成立** しなければならない
- したがって、実装は `AppContext.BaseDirectory` などの実行位置基準を優先し、`C:\MyApps\アプリ名` のような固定絶対パスをアプリ本体に埋め込んではならない
- 利用者がアプリホルダーを別ドライブ・別パスへ移した場合も、起動可能であることを正常系として扱う

---

## §C. 「保存先がアプリホルダー内」と「完結」は同義ではない

- ファイルがアプリホルダーにあるだけでは足りない
- そのファイルが **別 OS・別ユーザー・別マシンで復号不能** なら、「保存先は内側だが依存性は外側にある」
- DPAPI `LocalMachine` は「同一 Windows インスタンス」依存、`CurrentUser` は「同一 Windows ユーザー」依存である
- よって、実装者は各ファイルについて次を分類して記録する:
  - アプリホルダー内にあり、フォルダー移動にも耐えるもの
  - アプリホルダー内にあるが、OS/ユーザー依存を持つもの
  - System 側に状態が残るもの
- この分類を曖昧にしたまま「ホルダー完結」と表現してはならない

---

## §D. System 依存物の扱い

- SMB 共有、NTFS ACL、Firewall ルール、スケジュールタスク、ローカルユーザー、DPAPI 依存は、現実に Windows 側へ残る
- これらは Layer 1 違反ではなく、**機能上どうしても System に委ねる必要がある例外** として扱う
- ただし「例外だから放置」ではなく、**例外一覧として明示し、削除責任と復元責任を分けて設計する**
- 例外物は、少なくとも次を満たす:
  - 生成条件が説明できる
  - 削除方法が説明できる
  - 新規インストール前に掃除できる
  - 別 OS 側に残る可能性があるなら、その注意を明記する

---

## §E. インストーラーの責任(Layer 2)

- インストーラーは「ファイルを置く人」ではなく、「前の状態を片付けて、次の状態を整える人」である
- 新規インストールは、原則として **削除してから入れる**
- 更新インストールは、引き継ぐ対象を明文化したうえで **必要なものだけ確実に退避・復元する**
- 「更新で守る対象」の一覧が無いまま更新フローを作ってはならない
- アプリホルダー内の状態ファイルであっても、更新退避対象に漏れた時点で Layer 2 失敗である
- インストーラー作成時の修正は、**指定部分の変更ごとに毎回 Git へ保存する**。複数の直しをまとめて入れてはならず、テストと差し戻しができる単位で記録を残す
- Layer 2 は Layer 1 を信用しない
- したがって、削除対象の検出は登録情報だけでなく、実体確認、旧配置、旧保存先、周辺資産確認を含める
- `unins000.exe` 単独に依存してはならない
- `Uninstallable=no` を用いる場合、インストーラー自身が削除責任を全面的に負う

---

## §F. 更新インストールで守るべきもの

- 更新インストールは「バイナリ差し替え」ではない
- 更新インストールは「利用者が失いたくないものを守ったまま、新版へ移す」操作である
- よって、各アプリは更新前に少なくとも次を棚卸しする:
  - 設定
  - 秘密
  - 履歴
  - 保留データ
  - アイコンやカスタム画像
  - 識別情報
  - 証明書や通信の固定化情報
  - 利用者に見えにくい補助状態
- 「利用者に見えにくいから消えてもよい」としてはならない
- ShareWorkin で `userlist-state.json`、`swk-instance.json`、`notifycert.dat`、`icons` が更新退避対象から漏れていた事実を、再発防止の基準例とする

---

## §G. 削除の定義

- 「削除」とは、アプリホルダーを消すことではない
- 「削除」とは、利用者が後で困る残骸を残さないことである
- 削除対象は次を含む:
  - アプリホルダー
  - 旧保存先
  - 旧インストール先
  - スケジュールタスク
  - ローカルユーザー
  - SMB 共有
  - ACL 細工
  - 旧版移行のために一時的に参照した場所の残骸
- 実機で「削除後に何が残るか」を確認して初めて削除機能は完成とみなす

---

## §H. デュアルブート・複数 OS の考え方

- デュアルブート環境では、Windows 10 と Windows 11 は「同じ PC」ではあっても、System 状態は別である
- したがって、System 依存物の削除確認は OS ごとに必要である
- 「Windows 10 で消えたから Windows 11 でも消えているはず」と考えてはならない
- 共有、タスク、ローカルユーザー、Firewall、DPAPI 依存の一部は、OS インスタンス単位で確認する
- 基準書・運用メモは、この現実を前提に書く

---

## §I. 復元起点の設計

- 作業開始点を毎回きれいに作れることは、便利アプリのブランド品質に含む
- 復元起点は、原則として **共有ホルダー + アプリホルダーのバックアップ復元** で再現できるようにする
- System 依存物は、復元対象ではなく「起動時に再構築されるもの」として設計できるなら、その方が望ましい
- 逆に、System 側にしか存在せず再構築も説明不能な状態を増やしてはならない
- 復元設計では次を明文化する:
  - 何をバックアップするか
  - 何はバックアップ不要か
  - 何は起動時に再生成されるか
  - 何は同一 OS でしか復元できないか

---

## §J. 利用者から見える物の規律

- 利用者に見える場所には、意味のある物だけを置く
- 実装都合の副産物は可能な限り不可視にする
- ただし「不可視にしたから設計責任が消える」わけではない
- アプリホルダーに並ぶ物は、少なくとも開発者が「これは何のためにあるか」を一言で説明できなければならない
- 配布 ZIP に開発者向け検証物を混ぜない

---

## §K. 実装時の禁止事項

- アプリ本体に固定インストールパスを埋め込むこと
- 現役保存先として `AppData` を使うこと
- 更新インストール時の退避対象を明文化せずに実装すること
- `unins000.exe` 単独で「削除できる」と見なすこと
- System 依存物を「OS が持つものだから説明不要」と扱うこと
- 「今回はたまたま動いた」ことを基準適合と見なすこと
- 実行ファイル・インストーラーに発行元情報を埋め込まずに配布すること
- 利用者向けの説明責任情報（発行元・保存先・削除影響・更新元）を省略すること

---

## §L. 実装時の必須確認

- アプリ本体に絶対固定パスが残っていないか
- 保存先の一覧がアプリホルダー基準で説明可能か
- 更新インストールで守る対象が列挙されているか
- 新規インストール前の掃除対象に System 依存物が入っているか
- 削除後に残る物を実機で確認したか
- デュアルブートなら両 OS で確認したか
- バックアップ復元で再開できる起点が成立しているか
- 実行ファイルとインストーラーに発行元情報（会社名・URL）が埋め込まれているか
- Windows 10 / Windows 11 それぞれで起動時の警告動作を確認したか
- 利用者向けに確認導線（発行元・保存先・削除影響・更新元・問い合わせ先）が提供されているか

---

## §M. ゲートの責任

- 書面だけでは守られないため、ゲートを持つ
- ただしゲートは数を増やすことが目的ではない
- ゲートは、破ろうとした瞬間に「どの原則に反しているか」を示すために置く
- 優先してゲート化する項目:
  - 固定パス文字列の検出
  - `AppData` 等の現役保存先利用の検出
  - インストーラー内の更新退避対象漏れの点検
  - `Uninstallable=no` と削除ロジック整合の点検
- ゲートが無い間は、実機確認の責任がより重くなる

---

## §N. 協業運用

- AI は「書いてあった」ではなく、「守れているか」で評価する
- 指示者は、必要に応じて AI を役割分担させる
- 生成役と診断役を分けることは、責任逃れではなく品質確保の設計である
- ただし最終責任は、実装を出す側が持つ
- 基準書の改訂は、実装の失敗を隠すためではなく、再発の理由を潰すために行う
- 協業者が Codex であっても ClaudeCode であっても、**Git の基準はプログラムソース本体** である
- Git へ保存する対象は、原則として `ShareWorkin`、`ShareWorkin.SMB`、`ShareWorkinTray` と、それらを成立させる最小限のビルド定義・配布定義に限る
- `_works` は資料置き場であり、原則として Git 管理対象に含めない
- 調査用複製、レビュー退避、AI 作業用の staging、最小再現パッケージ等は、ソース本体と同列に扱ってはならない
- 協業者は、資料系・退避系・複製系のフォルダーを勝手に増やしてはならない。必要がある場合は、目的・保存要否・削除条件を先に明示する
- 協業者は、「現行コードにない」ことを理由に「過去にも存在しなかった」と断定してはならない。実機観測、配布物、Git 履歴、作業時の状況を分けて扱う
- 協業者は、意味不明な生成物・残骸・由来不明ファイルを直下やソース配下に残してはならない
- Git 保存の判断に迷う場合は、「これはプログラムソース本体か」「配布やビルドに必要な最小限か」「資料か残骸か」の順で確認し、基準外のものを本体管理へ混ぜない
- インストーラー、配布物、実機テストの再現確認に Git 番号を使う場合、その番号で追えるソースと成果物の対応関係を壊してはならない

---

## §O. 利用者への説明責任と信頼形成

- 利用者がインストールするアプリについて「誰が作ったか」「どこに保存されるか」「消したら何が消えるか」「更新はどこから来るか」を自分で確認できる状態を作ることは、動作品質と同じく制作責任の一部である
- 利用者が判断材料を持てる状態を作ることを、このブランドの説明責任とする

### インストーラーに含めるべき情報

- 発行元名（会社名または制作者名）
- サポート URL
- バージョン番号
- 更新情報への導線

### アプリ本体に含めるべき情報

- バージョン（ファイルプロパティ→詳細タブで確認可能）
- 発行元名・URL（同上）
- サポート先（同上）

### 利用者向けに提供すべき情報

- 保存場所（アプリホルダーの場所と内容）
- 削除対象（アンインストールで何が消えるか）
- 復元方法（バックアップから何がどこまで戻るか）
- 更新元（インストーラーの配布元）
- 問い合わせ先

### Windows 10 / Windows 11 の起動時警告への対応

- **Windows 10**: SmartScreen による警告が出るが、利用者が「詳細情報→実行」を選択することで判断できる
- **Windows 11**: Smart App Control が追加され、評価状態によっては利用者が「それでも実行」を選べない場面がある
- この差異はインストール案内文書で補足し、利用者が対処できるよう説明する
- 両 OS での起動時挙動確認を §L の必須確認に含める

## ShareWorkin から今回確定した教訓

- 「アプリホルダー完結」と言いながら Tray に固定絶対パスが残ることがある
- 「更新で引き継ぐ」と言いながら、状態ファイルの一部が退避対象から漏れることがある
- 「アプリホルダー内保存」と「フォルダーだけで完全復元」は別概念である
- 「Windows で消した」は、デュアルブートでは片系統しか消していない場合がある
- 「動作品質」を設計しながら「利用者への説明責任」を後回しにすると、必要な情報が製品に反映されないまま進む
- この種の漏れは、理念不足ではなく、確認単位の粗さから起こる

---

## 今回の到達目標

- 新規インストールで、理想状態から始められること
- 更新インストールで、守るべき状態が漏れないこと
- 削除で、利用者が後で困る残骸を極力残さないこと
- 復元で、共有ホルダー + アプリホルダーから再開できること
- 利用者が制作者・問い合わせ先・保存場所・削除影響・更新元を自分で確認できること
- これらを「ShareWorkin の特殊事情」ではなく、「便利アプリのブランドルール」として定着させること
