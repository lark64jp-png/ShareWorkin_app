# 0514 登録ユーザー誤認メモ

## 位置づけ

本件は Win10 / Win11 差分評価の前提不良。
先に「登録ユーザー誤認」を撲滅しないと、OS差ではなく誤認混入を見てしまう。

## 今回確認できた事実

- 本器 Win10 で `Win10_01.png` と `Win10_07.png` の時点で、`NEC` 行が `東芝店` として扱われていた。
- 表示だけでなく、`C:\MyApps\ShareWorkin\friends.json` の保存情報でも `NEC` が東芝側情報へ汚染されていた。
- 汚染後の `NEC` 保存値は以下だった。
  - `host = DESKTOP-DDEEF9N`
  - `share = 東芝店`
  - `lastKnownAddress = 192.168.0.218`
- その結果、NEC に対して東芝の証明書を照合しに行き、`TLS pinning mismatch` が発生した。

## 原因整理

- 原因1: 通知 TLS 証明書を再起動ごとに新規生成していた。
  - 再起動後に証明書サムプリントが変わり、保存済み証明書と不一致になった。
- 原因2: 自動観測結果で登録ユーザー本体の `host/share/IP` を更新していた。
- 原因3: 一致判定が緩く、`IP一致だけ` または `share名だけ` でも別PCへ寄る経路があった。

## この時点での修正

- 通知 TLS 証明書を `notifycert.dat` として永続化し、再起動で変わらないよう修正。
- 自動観測では登録本体の `host/share` を更新しないよう修正。
- `UserListWindow` / `MainWindow` の照合で、`IP一致だけ` / `share名だけ` の fallback を停止。
- 切替候補判定も `share名だけ` に限定し、`IP一致だけ` では候補化しないよう修正。

## 再テスト時の注意

- 現在の `friends.json` は一度汚染された可能性があるため、再テスト前に登録情報の健全性確認が必要。
- `登録ユーザー誤認` が再発しないことを先に確認し、その後に Win10 / Win11 差分へ進む。

## 今後の本修正方針

- 登録本体データと観測データを分離する。
- 観測結果では登録本体を書き換えない。
- 登録本体の変更は「明示的な再登録・接続先切替」のみ許可する。
- 一致しない観測結果は候補として隔離し、既存行へ混ぜない。
