# ShareWorkin 相談メモ

作成日: 2026-05-27

## 相談の目的

ShareWorkin の通知テスト不具合と、古いPC対応の考え方が混線しやすくなっているため、ChatGPT に対して次の2点を明確に相談したい。

1. 通知テスト不具合の切り分け手順が妥当か
2. 古いPCに合わせるとは何を意味するのか、どこまで対応すべきか

## 背景

- 通知テストまわりは今回で3回目の確認になっている
- 「修正したつもり」「見た目は改善した」「しかし別の条件で再発」の流れがあり、めどの付け方自体がずれている可能性がある
- 古いPCを用意して、低速環境でのもたつきや処理設計の問題を見つける方針でテストを始めている
- このアプリは個人のファイル転送機能として、使わなくなった古いPCを写真や資料のやり取り専用機として再活用できる可能性がある
- そのため、古いPCは単なる最低動作確認の対象ではなく、用途そのものの候補としても見ている

## いま起きている問題

### 1. 通知テストの不具合

- Windows通知は ON と認識しているログがある
- しかし `テスト通知を送る` を押しても反応しないケースがある
- ログでは `NotificationSettings.SendTestNotification requested` の直後に、パイプ切断系のエラーが出ていた
- そのため、見た目の「通知状態」と実際の通知送信成功は一致していない

### 2. 表示と実体のズレ

- 画面上の色や文言が、OS通知状態だけでなく「未確認」など他条件を混ぜて判定している
- そのため、OS通知が ON でも、ユーザーには OFF のように見える場面がある
- この問題は通知送信そのものの不具合とは別で、表示設計の問題でもある

### 3. 古いPC対応の不安

- 古いPCでは、遅さそのものは避けられない部分もある
- ただし「遅いだけ」なのか「処理設計が悪くて止まって見える」のかは分けて考えたい
- 古いPC向け最適化を進めると、通常PCでの設計や保守性を崩すおそれもある
- ただし、古いPCを専用機として活かせるなら、多少の性能差を前提にした別の価値設計もありうる

## これまでに分かったこと

### 通知まわり

- OS通知ON/OFFはログで把握できる
- 通知テストの成功可否も、ログ設計上は本来追えるようにしている
- ただし、修正前ログでは UI から Tray への通知テスト送信経路が壊れていた可能性が高い

### 最新ログで追加確認できたこと

- `2026-05-27 10:49:09` 時点で `windowsNotificationsEnabled=True`
- 1回目のテスト通知押下 (`10:49:13`) は `UiPipeClient.SendNotificationCommand skipped: not connected` で失敗
- 2回目のテスト通知押下 (`10:49:19`) は
  - `TrayPipeServer TEST_NOTIFICATION received`
  - `NotificationSettings.SendTestNotification acknowledged by tray`
  まで進み、実際に通知が表示された
- その直後に `Unobserved task exception` として `UiPipeClient.Connect` 起点の `NullReferenceException` が出ている
- ユーザー観測では「通知が出たと同時にボタンの点灯がなくなった」

### この結果が意味すること

- 通知経路は完全に死んでいたわけではなく、再試行で成功する状態まで改善している
- 一方で、接続管理まわりにまだ不安定さが残っている
- 「通知が出た」ことと「ボタン色が消えた」ことは、同じタイミングで起きていても別レイヤーの問題である可能性が高い
- とくに未監視例外が出ているため、成功後に UI 側の状態更新や接続状態保持が崩れている疑いがある

### 一覧表示まわり

- FriendShop の再読込で一覧を空にしてから再表示していたため、遅いPCでは「一瞬消える」「選択が飛ぶ」ように見えやすかった
- 現在は、次の内容を組み立ててから差し替える方向へ修正した
- 「表示状態が変わっていないなら選択維持、変わったらクリア」という整理を進めている

## 迷走しやすいパターン

### パターンA: 表示改善を先にやって実機不具合の確認が遅れる

- ボタン色や文言を直す
- しかし通知送信経路そのものが壊れていた
- 結果として、安心できる材料が増えず、確認回数だけ増える

### パターンB: 古いPCの遅さを全部「不具合」と見てしまう

- 低速環境では一定の待ち時間はありうる
- ただし「旧表示を残す」「処理中を示す」などで回避できるUX問題もある
- 遅いこと自体と、設計不備による誤操作誘発は分離すべき

### パターンC: 1つの表示に複数の意味を持たせる

- 例: ボタン色が OS通知状態、未確認状態、通知モードOFF を全部兼ねてしまう
- ユーザーは何が原因で注意色なのか判別しにくい

## いま必要な整理

### 優先順位

1. 通知テストが実際に成功するか
2. 修正後ログで成功フローを確認できるか
3. そのうえでボタン色や文言の条件を見直す
4. 古いPCでの体感差を、仕様上許容か設計見直しかに分類する

### 「直った」とみなす条件

- テスト通知ボタン押下で UI が止まって見えない
- ログに通知送信要求、Tray受信、ACK の流れが出る
- 実際に Windows 通知が表示される
- 再現性があり、1回だけでなく複数回成功する

## 古いPC対応について相談したいこと

### 相談したい観点

- 古いPCに合わせるとは、どこまでを目標にするべきか
- 「多少遅いが安全に使える」と「十分軽い」は分けるべきか
- 低速環境では、性能改善よりも「途中で消さない」「操作不能に見せない」「再入力を誘発しない」設計を優先すべきか
- 古いPCをやり取り専用機として再活用する方向へ、アプリの価値提案を広げるべきか

### 現時点の仮説

- 古いPC対応の第一目標は、速度そのものより誤操作防止
- 具体的には次が重要

- 表示更新中に一覧を消さない
- 選択中項目を不用意に飛ばさない
- 同期待ちで画面が固まって見える操作を減らす
- いま何を待っているか分かるようにする

## ChatGPT に聞きたい質問

1. 通知テスト不具合の切り分け順序として、先に実送信成功を確認し、その後で表示条件を整理する進め方は妥当か
2. OS通知状態、ShareWorkin通知モード、通知テスト確認済み状態を、UI上でどう分離表示するのが分かりやすいか
3. 古いPC対応は「処理を速くする」よりも「途中状態で誤解させない」に重点を置くべきか
4. 低速環境を基準にする場合、どの症状を「要修正」、どの症状を「許容範囲」と整理すべきか
5. 同じテーマで調査が長引くとき、迷走を防ぐために毎回固定で確認すべきチェック項目は何か
6. このアプリを、使わなくなった古いPCを写真・資料のやり取り専用機として再活用できる方向へ進める価値はあるか
7. その方向へ進めるなら、通常PC向け設計を崩さずに両立するには何を分けて考えるべきか

## 補足

- 通知テスト不具合とボタン色問題は別件
- ただし、ユーザー体験上は連続して見えるため、説明順序を誤ると混乱しやすい
- 今回は「機能を直す」「表示を整える」「古いPC対応を考える」の3本を混ぜずに扱いたい

## ChatGPT への伝文案

今回の最新ログでは、Windows通知は ON 判定です。通知テストは1回目が未接続で失敗しましたが、2回目では Tray 側受信と ACK まで確認でき、実際に通知も表示されました。その一方で直後に `UiPipeClient.Connect` 起点の `NullReferenceException` が未監視例外として出ており、ユーザー側では「通知が出たと同時にボタンの点灯がなくなった」と見えています。つまり、通知送信自体は一部成功したが、接続管理またはUI状態更新がその後に崩れている可能性があります。また、このアプリは使わなくなった古いPCを写真や資料のやり取り専用機として再活用する方向も考えられるため、古いPC対応を単なる最低性能対応として扱うべきか、用途拡張として捉えるべきかも相談したいです。
