# 利用者特定仕様_2026-05-26

## 目的

ShareWorkin における「利用者特定」は、誤認して別の登録利用者名を表示しないことを最優先とする。

本仕様は、今後の実装・テスト・ログ確認の基準とする。

## 基本方針

1. ShareWorkin は SMB 共有そのものを本人証明とはみなさない。
2. SMB 共有は「ファイルが届く経路」であり、「誰が操作したか」を単独では保証しない。
3. 利用者特定は、ShareWorkin 独自の通知経路と登録情報の照合を優先根拠とする。
4. 確信が足りない場合は、登録利用者名を出さず `未照合` または `不明アクセス` とする。
5. 誤認して別の登録利用者名を出すより、不明として扱うことを正しい挙動とする。

## 用語定義

### 1. 登録利用者

ShareWorkin の Friend 登録に存在する相手。

保持情報の主な例:

- 表示名
- HostMachineName
- ShareName
- RemoteSwkInstanceId
- OwnerCertThumbprint
- LastKnownAddress
- LastShareAccessVerifiedAt
- LastShareAccessSwkInstanceId

### 2. 正規アプリ間共有

次の両方を満たす受信または操作。

- ShareWorkin の通知経路が存在する
- その通知内容が登録利用者と照合成功する

### 3. 未照合アプリ共有

ShareWorkin の通知経路は存在するが、登録利用者と照合できない受信または操作。

### 4. 不明アクセス

SMB 上の変化は観測されたが、ShareWorkin の通知経路で正規共有と確認できない受信または操作。

これは以下を含む。

- 通知なしでの受信
- 通知はあるが照合根拠不足
- 照合候補が複数あり、1人に確定できない

## 特定レベル

ShareWorkin は、利用者特定を次の 3 段階で扱う。

### レベルA: 正規アプリ間共有

条件:

- 通知あり
- `senderSwkInstanceId` または `Host + Share` が登録利用者と一意に一致

扱い:

- 登録利用者名を表示してよい
- 履歴上も登録利用者として扱ってよい
- 通常の受信・交流として扱う

### レベルB: 未照合アプリ共有

条件:

- 通知あり
- ただし登録利用者と一意に照合できない

扱い:

- 登録利用者名を表示してはいけない
- `未照合` として表示する
- 原因調査用のログを残す

### レベルC: 不明アクセス

条件:

- 通知なしで SMB 上の変化だけが観測された
- または通知があっても正規共有として扱う基準を満たせない

扱い:

- 登録利用者名を表示してはいけない
- `不明アクセス` または `送信元未特定` として扱う
- 更新履歴・通知履歴・ログに残す

## 判定優先順位

利用者照合は、次の順で行う。

1. `RemoteSwkInstanceId` と `senderSwkInstanceId` の一致
2. `HostMachineName + ShareName` と `senderMachineName + senderShareName` の一致
3. 共有到達確認済み情報との整合
4. それでも確定できなければ未照合または不明アクセス

## 禁止事項

次の判定はしてはならない。

1. `Host` だけ一致したから同一利用者とみなす
2. `ShareName` だけ一致したから同一利用者とみなす
3. SMB 接続できたから登録利用者本人とみなす
4. 通知が来たから自動的に登録利用者本人とみなす
5. 画面に見えている live shop だけで登録利用者を確定する
6. 誤認の可能性がある状態で登録利用者名を履歴へ残す

## SMB 共有の位置づけ

SMB 共有は ShareWorkin の土台ではあるが、本人確認の根拠ではない。

したがって、同一 PC から Windows 共有として直接ファイル操作された場合、ShareWorkin はそれを自動的に「登録利用者本人の操作」とはみなさない。

今後の仕様上は、次のように分ける。

- ShareWorkin 通知と照合できた操作: 正規アプリ間共有
- SMB のみで確認された操作: 境界外アクセス候補

## 表示仕様

### 正規アプリ間共有

- 登録利用者名を表示する
- 通常の受信表示を行う

### 未照合アプリ共有

- `未照合`
- `送信元を確認できない交流通知`
- `通知は来たが相手未特定`

のいずれかの文言で表示する。

### 不明アクセス

- `不明アクセス`
- `送信元未特定の受信`
- `通知なし受信`

のいずれかの文言で表示する。

更新履歴では、特別確認対象として通常警告色とは別の背景帯で扱う。

## 履歴仕様

### 更新履歴

- 正規アプリ間共有: 登録利用者名付きの受信履歴
- 未照合アプリ共有: 未照合受信履歴
- 不明アクセス: 不明アクセス履歴

### 通知履歴

- 通知あり受信
- 通知あり未照合
- 通知なし受信

を識別できることが望ましい。

## ログ仕様

未照合または不明アクセス時は、後追い調査のため次を残す。

1. eventId
2. senderMachineName
3. senderShareName
4. senderSwkInstanceId
5. receiverShareName
6. targetName
7. targetFullPath
8. sourceRoute
9. 一致候補数
   - instance 一致候補数
   - host+share 一致候補数
   - host 一致候補数
   - share 一致候補数
10. 通知あり / なし
11. 未照合になった理由

## テスト方針

今後のテストでは、次を重点確認とする。

1. 正常系で正しく登録利用者名になるか
2. 長時間放置後も誤認せず維持できるか
3. 再起動後に誤認せず維持できるか
4. IP 変化や再接続後に誤認しないか
5. 照合根拠が崩れたとき、別人名ではなく未照合に落ちるか
6. 通知なし受信が不明アクセスとして残るか

## 今後の実装指針

1. 正規アプリ間共有を最優先で扱う
2. SMB 単独観測は本人確定に使わない
3. 未照合と不明アクセスを明確に区別する
4. 画面表示・履歴・ログの分類を揃える
5. 誤認防止を、利便性より優先する

## 本仕様の結論

ShareWorkin における利用者特定は、「人そのものの本人確認」ではなく、「登録済み相手環境と ShareWorkin 通知経路を組み合わせた整合確認」として扱う。

そのため、ShareWorkin は次を正式方針とする。

- 通知と照合できたアクセスだけを正規共有として扱う
- それ以外は無理に登録利用者へ結びつけない
- 未照合または不明アクセスとして記録し、利用者へ可視化する

この方針を、以降の設計・修正・テストの基準とする。
