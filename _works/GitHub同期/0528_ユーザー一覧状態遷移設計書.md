# ShareWorkin ユーザー一覧 — 状態遷移設計書

作成日：2026-05-28
ステータス：設計案（実装未着手）

---

## ■ 設計の目的

現在のユーザー一覧は、状態・候補・操作提案・暫定判定が混在しており、
「今なぜその表示になっているか」が利用者にも開発側にも分かりにくい。

今回の再設計で達成したいこと：

- 「接続不明」へ即落とさず、その前に正式な猶予状態を置く
- 状態と操作提案を分離する
- 各状態の入口・出口・ログを明確にする
- 自動復旧と手動確認の境界を明文化する

---

## ■ 状態一覧と位置づけ

```
【登録済み Friend の状態系列】

  接続中
    ↕（スキャン毎に維持 or 離脱）
  接続確認中
    ↕（共有検証の成否）
  ──────────────── 接続喪失ライン ────────────────
  自動復旧中         ← 猶予1：即断定しない
    ↓（リトライ上限）
  一時不安定         ← 猶予2：待機・再試行
    ↓（時間窓超過）
  接続不明           ← 最終確定（候補なし）
    or
  手動確認待ち       ← 最終確定（候補あり・証明書問題・SwkID変更）
    or
  要更新             ← 相手PC再インストール等で SwkInstanceId が変わった場合

【未登録 の状態系列】

  未登録             ← LAN上に ShareWorkin が見える・Friend 登録なし
```

## ■ 各状態 詳細シート

---

### 状態 1：接続中

| 項目 | 内容 |
|---|---|
| **入口条件** | ① liveShop が見つかっている（SwkInstanceId 一致 or ホスト+共有一致） AND ② `FriendShareAccessTracker.IsVerifiedFor()` が true AND ③ `HasCertificateMismatch` が false |
| **自動処理** | スキャン間隔ごとに liveShop の継続を確認。変化があれば `UpdateFriendExternalState()` で記録更新 |
| **表示文言** | `接続中` |
| **表示色** | 緑 `#4CAF50` / 背景 `#E8F5E9` |
| **表示ボタン** | なし（または「開く」のみ） |
| **離脱条件** | liveShop が 1回でも消えたら → **自動復旧中** へ |
| **ログ** | `[UserList] Connected: FriendId={id}, Host={host}, SwkInstanceId={id}` |

---

### 状態 2：接続確認中

| 項目 | 内容 |
|---|---|
| **入口条件** | ① liveShop が見つかっている AND ② `IsVerifiedFor()` が false（共有接続がまだ確認されていない） AND ③ `HasCertificateMismatch` が false |
| **自動処理** | バックグラウンドで `TryVerifyRegisteredShare()` を実行。成功すれば **接続中** へ |
| **表示文言** | `接続確認中` |
| **表示色** | 橙（薄）`#BF5700` / 背景 `#FFF3E0` |
| **表示ボタン** | `「再開」`（手動で共有確認をトリガー） |
| **離脱条件** | 共有確認成功 → **接続中** ／ liveShop 消失 → **自動復旧中** |
| **ログ** | `[UserList] VerifyPending: FriendId={id}, liveShop={host}` |

---

### 状態 3：自動復旧中

| 項目 | 内容 |
|---|---|
| **入口条件** | ① liveShop が見つからない AND ② `LastFoundAt` が存在する（過去に接続実績あり） AND ③ リトライカウント < 3 |
| **自動処理** | スキャン間隔ごとに `ResolveLiveFriendShopWithRetry()` を実行。見つかれば **接続確認中** へ。3回失敗で **一時不安定** へ。カウントを `_retryCount` として保持（メモリ内でよい） |
| **表示文言** | `接続復旧中…` |
| **表示色** | 橙 `#BF5700` / 背景 `#FFF3E0` |
| **表示ボタン** | なし（自動処理中を示すインジケーターのみ） |
| **離脱条件** | liveShop 発見 → **接続確認中** ／ 3回失敗 → **一時不安定** ／ cert-mismatch 検出 → **手動確認待ち** ／ 別SwkInstanceId 検出 → **要更新** |
| **ログ** | `[UserList] AutoRecovery: FriendId={id}, attempt={n}/3` ／ `[UserList] AutoRecovery failed at attempt={n}` |

> **補足：** スキャン間隔は現在8秒なので、自動復旧中の最大猶予は約24秒（3回 × 8秒）。起動直後・短時間のネットワーク揺れはここで吸収される。

---

### 状態 4：一時不安定

| 項目 | 内容 |
|---|---|
| **入口条件** | ① 自動復旧中のリトライが3回失敗 AND ② `FirstMissAt` から5分未満 |
| **自動処理** | 次スキャンでも liveShop 探索を継続（ただし静かに）。見つかれば **接続確認中** へ。5分経過で分岐判定へ |
| **表示文言** | `一時不安定` |
| **表示色** | 橙（薄）`#BF5700` / 背景 `#FFF3E0` |
| **表示ボタン** | `「再スキャン」`（手動でスキャンを即トリガー） |
| **離脱条件** | liveShop 発見 → **接続確認中** ／ 5分超過 → **分岐判定**（接続不明 / 手動確認待ち / 要更新） ／ cert-mismatch → **手動確認待ち** |
| **ログ** | `[UserList] Unstable: FriendId={id}, missingSince={t}` ／ `[UserList] Unstable resolved: FriendId={id}` |

> **補足：** 「5分」はネットワーク復帰・名前解決待ちをカバーする最低ライン。調整可能な定数として定義すること。

---

### 状態 5：接続不明

| 項目 | 内容 |
|---|---|
| **入口条件** | ① 5分超過 AND ② LAN上に候補なし AND ③ cert-mismatch なし AND ④ SwkInstanceId 変更なし |
| **自動処理** | 自動処理なし。次のスキャンでも探索は続けるが、状態ラベルは変えない（見つかれば **接続確認中** へ静かに復帰） |
| **表示文言** | `接続不明` |
| **表示色** | 赤 `#963228` / 背景 `#FFEBE6` |
| **表示ボタン** | `「スキャン」`（手動での全体スキャンを促す） |
| **補足メモ** | 「相手のPCがオフラインか、ネットワークが変わった可能性があります。」 |
| **離脱条件** | 次スキャンで liveShop 発見 → **接続確認中** ／ 候補が新たに現れた → **手動確認待ち** |
| **ログ** | `[UserList] Unreachable: FriendId={id}, missingSince={t}, reason=no-candidate` |

---

### 状態 6：手動確認待ち

| 項目 | 内容 |
|---|---|
| **入口条件（いずれか）** | A: 5分超過 AND 候補あり（共有名一致 or SwkInstanceId なし相手） ／ B: `HasCertificateMismatch = true` ／ C: ユーザーが操作をキャンセルして戻った |
| **自動処理** | なし。利用者のアクション待ち |
| **表示文言** | `手動確認待ち`（サブ理由を付記） |
| **表示色** | 赤 `#963228` / 背景 `#FFEBE6` |
| **表示ボタン** | `「確認する」`（FriendsWindow を開く） |
| **補足メモ（理由別）** | A: `候補：{host}/{share}。同じ相手か確認してください。` ／ B: `通知経路に問題があります。接続情報を再確認してください。` |
| **離脱条件** | 利用者が FriendsWindow で操作確定 → **接続確認中** ／ 操作キャンセル → 状態維持 |
| **ログ** | `[UserList] ManualRequired: FriendId={id}, reason={cert-mismatch\|candidate\|user-cancel}, candidate={host}` |

---

### 状態 7：要更新

| 項目 | 内容 |
|---|---|
| **入口条件** | ① `RemoteSwkInstanceId` が記録されている AND ② 同じホスト名/IPで**別の SwkInstanceId** が出現 AND ③ cert-mismatch なし |
| **自動処理** | なし。相手のPCが再インストールされた可能性があるため、自動更新はしない |
| **表示文言** | `接続先更新あり` |
| **表示色** | 橙 `#BF5700` / 背景 `#FFF3E0` |
| **表示ボタン** | `「再接続」`（招待コードを交換して再登録） |
| **補足メモ** | `同じ場所に別の接続情報が見つかりました。再接続して確認してください。` |
| **離脱条件** | 利用者が再接続操作完了 → **接続確認中** ／ 操作キャンセル → 状態維持 ／ 相手が消えたら → **接続不明** or **一時不安定** |
| **ログ** | `[UserList] RelinkCandidate: FriendId={id}, oldSwkId={old}, newSwkId={new}, host={host}` |

---

### 状態 8：未登録

| 項目 | 内容 |
|---|---|
| **入口条件** | LAN上の ShopInfo に Friend 登録のないエントリーが存在する |
| **自動処理** | なし |
| **表示文言** | `未登録` |
| **表示色** | 黄橙 `#FF9800` / 背景 `#FFF8E1` |
| **表示ボタン** | `「登録する」` |
| **離脱条件** | 利用者が登録操作完了 → **接続確認中** ／ 相手が消えたら → 行ごと消える |
| **ログ** | `[UserList] NewShop: host={host}, share={share}, swkId={id}` |

## ■ リトライ回数・時間窓の設計意図

| パラメーター | 提案値 | 根拠 |
|---|---|---|
| 自動復旧リトライ上限 | **3回** | スキャン間隔 8秒 × 3 = 約24秒。起動直後の遅延・短時間揺れを吸収 |
| 一時不安定の時間窓 | **5分** | ネットワーク復帰・DHCP再取得・名前解決を待つ最低ライン |
| 定数名（提案） | `MaxAutoRecoveryAttempts = 3` ／ `UnstableWindowMinutes = 5` | 調整可能な定数として分離 |

> 両方とも現在コードにはなく、導入時に追加が必要な値です。

## ■ 「自動遷移」（UserListWindow 自動起動）の実態

現行コードでは、リロードボタンからではなく、
**エクスプローラーの再接続フロー**から UserListWindow が自動起動される。

`MainWindow.xaml.cs` の `ExecuteRefreshAsync()` 末尾：

```
① キャッシュ上の liveShop 確認
   ↓ なし
② 現在パスへの直接アクセス確認
   ↓ 失敗
③ MachineName で UDP キャッシュ再探索 → 見つかれば自動復帰
   ↓ なし
④ ResolveLiveFriendShopWithRetry（クイックスキャン）→ 見つかれば自動復帰
   ↓ なし
⑤ ★自動で UserListWindow を開く  ← L2121
```

体験上「リロードしたらユーザー一覧が出た」ように見えるのは、
①〜④の自動復帰がすべて失敗した直後に ⑤ が走るため。
これは仕様として維持する（状態遷移設計と矛盾しない）。
