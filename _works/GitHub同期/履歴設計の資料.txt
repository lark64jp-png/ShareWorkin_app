履歴設計の資料

目的
履歴機能を「ユーザー別履歴」「自分中心の全体履歴」「通知範囲」に分けて整理する。

1. 全体方針

- ユーザー情報更新画面は「その相手との関係履歴」を扱う
- トップページ下部は「自分を基点にした全体履歴」を扱う
- 通知範囲は、アプリ側で設定できる範囲と、Windows 側に依存する範囲を分けて考える

2. ユーザー情報更新画面

位置づけ
- ここは「そのユーザーとの履歴」に絞る
- 相手ごとの関係性を確認する画面にする

右側3枠の意味
- わたしのお店の許可状況
  その相手に対して、現在どういう共有条件かを見る欄
- お友達からのアクセス履歴
  相手から自分に来た履歴
- お友達へのアクセス履歴
  自分から相手へ見に行った、または渡した履歴

この画面で持つ履歴項目案
- 日時
- 種別
  もらう / 渡す / 接続 / 切断 / 再開 / 失敗
- 相手名
- 対象
  ファイル名またはフォルダー名
- 結果
  成功 / 失敗
- 補足
  再開待ち / 相手不在 / 証明書不一致 など

表示文言の考え方
- 業務ログ風にしすぎず、一般利用者が読んで分かる自然文にする

表示例
- 2026/05/15 07:12 東芝さんから 001.pdf を受け取りました
- 2026/05/15 07:14 東芝さんのお店に入りました
- 2026/05/15 07:16 接続を再開できませんでした

3. トップページ下部

位置づけ
- ここは「自分の履歴」
- 相手は全員対象
- 自分を基点にして、全体の出来事を把握する

各ボタンの意味
- アクセス履歴
  自分と他ユーザーの出入り全般
- 更新履歴
  自分のお店の中で起きた変更
  追加 / 受取 / 移動 / 削除 / 名前変更 / 共有状態変更
- 通知履歴
  実際に BK アプリが通知対象として扱ったもの

見分け方の考え方
- 同じ出来事でも、役割ごとに見え方を分ける

例
- 「東芝さんからファイルを受け取った」
- アクセス履歴: 誰とのやり取りか
- 更新履歴: 自分のお店に何が増えたか
- 通知履歴: 通知を出したかどうか

4. 通知範囲

現状の整理
- 現状コード上でも、通知範囲はアプリ側で制御できる
- 選択肢は「すべて / 外部のみ / なし」

アプリ側で制御できる範囲
- すべて
- 外部のみ
- なし

補足
- アプリが「通知を出すかどうか」は制御できる
- ただし、通知が必ず表示されるかは Windows 側設定にも依存する

Windows 側要因
- Windows の通知オフ
- 集中モード
- 通知センター側の抑制

仕様上の考え方
- 「通知を出す設定」まではアプリ責務
- 「必ず表示される保証」までは OS 依存

5. 初期実装案

最初は欲張らず、以下の範囲で始める

ユーザー別履歴
- もらう
- 渡す
- 接続
- 再開失敗

全体履歴
- アクセス履歴
- 更新履歴
- 通知履歴

通知範囲
- すべて
- 外部のみ
- なし

保存方針
- 保存件数は 100〜300 件程度から開始
- 保存期間は無期限にせず、上限件数で古いものから落とす

6. 今後の拡張余地

- 期間検索
- 相手別絞り込み
- 成功 / 失敗だけ表示
- 詳細理由の表示
- 画面ごとの見せ方調整

7. 打ち合わせで確認したいこと

- ユーザー別履歴に「切断」を含めるか
- 更新履歴に「削除」を見せるか
- 通知履歴は「通知対象になった記録」なのか「実際に出した記録」なのか
- 保存件数の初期値を 100 件にするか 300 件にするか
- 一般利用者向けの文言をどこまでやさしくするか
