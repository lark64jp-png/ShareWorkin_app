## 1. この文書の目的

更新履歴の見え方を良くするだけでは足りない。  
その前段として、**止めるべき操作を止める判定** が必要である。

特に ShareWorkin では、

- 何を許すか
- 何を止めるか
- なぜ止めたか
- 止めたあと何を代替手段として考えるか

が曖昧なままだと、履歴も曖昧になる。

この文書は、実装より一段手前で

- どの NG 判定が不足しているか
- なぜそれが不足なのか
- 次にどこから埋めるべきか

を整理するために作る。

加えて、この文書は単なる論点メモではない。  
ShareWorkin のエクスプローラー機能からアプリ完成まで、

- 何を大事にし続けるか
- 何を曖昧にしないか
- どこで見せ方より実体保護を優先するか

を忘れないための文書でもある。

## 2. 今回見えた中心論点

### 2-1. NG判定は失敗ではなく露出面である

今回の対話で重要だった考え方:

- NG 判定が露出することで、対応手段を考えられる
- 止めたこと自体が、仕様の未整理部分を照らす
- NG 履歴は「できなかった記録」ではなく、「ここに設計判断が要る」という観測点になる

したがって、NG を出すことは後退ではない。  
むしろ、ShareWorkin の履歴を賢くするために必要な工程である。

### 2-2. 保留ホルダーの定義が鍵

西村さんの整理:

- 保留は余剰在庫、売値未定品、シーズン在庫など用途は多岐に渡る
- しかし **非公開ホルダー** である
- 店の内部運用上は必要だが、通常棚と同列ではない

このため、保留は

- 単なる別フォルダーではない
- 通常移動先と同列に扱うべきではない
- バックヤード的意味を持つ

と解釈するのが自然である。

### 2-3. この課題はテスト観点ではなく品質方針でもある

今回の課題は「この基準でテストできるか」だけでは足りない。  
むしろ重要なのは、

- NG〜Warning 判定を完成まで意識し続けること
- 更新履歴を品質判断の中核として扱うこと
- 実装・運用・説明のどこで答えるべきかを分けること

である。

このため、本課題は単発の試験設計ではなく、ShareWorkin 全体の品質方針として扱う。

## 3. 今回すでに入れた実装的対応

このセッションで先行して入れた対応:

1. 通常の移動先ダイアログから `保留` を外した
2. 通常 `Move` で保留へ入ろうとすると `Blocked`
3. 通常 `Copy` で保留へ入ろうとしても `Blocked`
4. 上位アクセス権の範囲内での移動では、余計な権限変更と余計な共有設定変更履歴を出さないよう修正した

これは、「保留は通常公開棚と同列ではない」「意味のない権限変更を増やさない」という定義を、まず UI とサービス層へ反映した第一歩である。

## 4. まだ不足している NG 判定課題

### 4-1. 最優先: 複数選択 Ctrl+ドラッグ の抜け道

現状の大きな懸念:

- 複数選択の `Ctrl+ドラッグ` は、単体コピーとは別経路に落ちる
- その結果、`CopyItem` 側の NG 判定を通らない可能性がある
- 保留流入禁止や配下コピー禁止が、経路によっては抜けるおそれがある

これは優先度が高い。

### 4-2. 保留配下からの通常操作の整理不足

未整理:

- 保留の中のものを通常 `Move` で別棚へ出してよいか
- 保留の中のものを通常 `Copy` で増やしてよいか
- 保留の中のものをさらに保留へ送ることに意味があるか

ここは業務ルールとして明文化が必要。

### 4-3. Hold 操作自体の禁止条件不足

候補:

- 保留配下から保留への再保留
- 保留先異常時の事前ブロック
- 意味のない経路や同義反復操作の禁止

現在は UI 側の流れである程度防いでいるが、サービス層ルールとしてはまだ弱い。

### 4-4. Delete の保護ルール不足

現在の削除は、存在確認中心である。  
業務上「消してはいけない対象」が増えた場合に、サービス層で守れない。

候補:

- 保留そのものの削除禁止
- 今後保護対象が増えたときの共通判定

## 5. 課題定義としての保留ホルダー

保留ホルダーの課題は、単なる UI の細部ではない。  
少なくとも次の5課題に分かれる。

### 5-1. 役割定義の課題

- 保留は「内部運用領域」かつ「非公開」
- 通常棚と同列に扱わない

### 5-2. 操作ルールの課題

- 保留へ入れる操作
- 保留から戻す操作
- 保留内で許す操作 / 止める操作

を分けて考える必要がある

### 5-3. レポート設計の課題

履歴は単なる `Move` ではなく、

- 非公開側へ退避した
- 非公開側から戻した
- 非公開なので通常移動を止めた

と読める必要がある

### 5-4. 実装境界の課題

- UI で止めているだけでは弱い
- `ExplorerActionService` 側でも業務ルールとして持つ必要がある

### 5-5. 到達条件の課題

アプリ上で見える共有状況は、設定や内部状態を読み上げて変換した一時表現である。  
したがって、保留ホルダーについて本当に保証すべきことは、

- アプリ上で見えないこと

ではなく、

- Windows 共有の実体として共有相手が到達できないこと

である。

このため、保留ホルダーは通常フォルダーと同じ共有設定評価で扱わず、**非到達条件で管理する特別領域** として扱う。

## 6. 次に決めたいルール

### 問い1: 保留配下からの通常 Move / Copy をどうするか

選択肢:

- `A`: 許す。ただし保留から店頭へ戻すのは専用操作を基本にする
- `B`: 原則 NG。保留から出すのは「お店に戻す」中心にする
- `C`: 移動は許すが、コピーは NG

Codex の現時点推奨: `B`

理由:

- 非公開バックヤードという定義に最も合う
- 履歴上の意味がぶれにくい
- NG 判定と代替導線が整理しやすい

### 問い1の整理後結論

- 通常フォルダーは、上位階層以上の許可設定を出さない
- 保留フォルダーは、通常共有評価で扱わない
- 保留フォルダーの基準は、共有相手から到達できないこと
- この条件をプログラムで担保できるなら、保留は共有 OFF 相当として扱ってよい

したがって、保留は通常棚の延長ではなく、共有露出対象外の内部運用領域として扱う。

### 問い2: NG履歴に何を持たせるか

今後必要になりやすい情報:

- 止めた理由
- 移動元
- 移動しようとした先
- 代わりに使うべき操作のヒント

これは履歴側だけではなく、結果モデル側の器も強くする必要がある。

### 問い3: 削除は本当に削除として扱うか

クラウドストレージと PC ローカル保存の期待差を踏まえると、この問いは単なる削除実装の話ではない。  
ShareWorkin のエクスプローラーが利用者に何を意味するかの問題である。

整理:

- 完全削除をアプリ上の `削除` と呼ぶと、誤操作時の意味が重くなりすぎる
- 逆に、削除と言いながら曖昧な退避をすると、保留との役割がぶれる
- 仕様説明として正しい意味を持たせる必要がある

結論:

- ShareWorkin のエクスプローラー上での削除は、完全削除ではなく **ごみ箱への移動** として扱う
- 完全削除はアプリ外、または別責任の操作として分ける
- 保留は業務上の内部運用領域であり、ごみ箱とは役割を分ける

この方針により、利用者説明、履歴、運用ルールの一貫性を保てる。

## 7. 直近の具体提案

### 提案1

複数選択 `Ctrl+ドラッグ` の経路を見直し、単体コピーと同じ NG 判定を通す。

### 提案2

保留配下からの `Move` / `Copy` ルールを確定し、それに応じた `Blocked` をサービス層へ入れる。

### 提案3

`Blocked` 履歴の情報密度を上げる。  
少なくとも「なぜ止めたか」「どこへ行こうとしていたか」が一覧または詳細で追える形に寄せる。

### 提案4

保留ホルダーを「共有相手から非到達であること」で扱うため、UI 表示ではなく Windows 共有実体側の保証条件を点検対象へ上げる。

### 提案5

アプリ上の `削除` を、ごみ箱移動として意味づけ直す。  
そのうえで、保留とごみ箱の役割を混ぜずに履歴文言と導線を整える。

## 9. 今回の明示結論

今回の対話と実機修正確認を踏まえ、現時点の明示結論を次の通り置く。

1. NG〜Warning 判定は、ShareWorkin の品質方針として完成まで持ち続ける
2. 更新履歴は、単なる表示補助ではなく品質判断の中核機能として扱う
3. 通常フォルダーは、上位階層以上の許可設定を出さない
4. 保留フォルダーは、通常共有評価ではなく非到達条件で扱い、共有 OFF 相当としてよい
5. アプリ上の削除は、完全削除ではなくごみ箱移動として扱う
6. 保留とごみ箱は役割を分ける

## 10. 一言まとめ

今回の主題は、「履歴を見やすくする」ことではない。  
**NG 判定を露出させることで、ShareWorkin の運用ルールそのものを言語化し、履歴に意味を持たせること** が主題である。

特に保留ホルダーと削除機能は、今後の Explorer / 履歴 / 共有ルールの交点になる。  
ここを、

- 保留は非公開の内部運用領域
- 削除はごみ箱移動

として言い切れることが、次の整理の起点になる。
